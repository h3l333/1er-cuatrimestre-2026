>Main and "official" identifier of a table. Out of all the *candidate keys* ([[Claves candidatas]]), one is chosen so that the system can associate the table in question with other tables within the database schema. A primary key can be considered a candidate key that has been "promoted".

### Primary key selection criteria
- Usage of space
- Key functionality within the database system

---

A primary key is stored repeatedly across the database because:

- It appears in the table where it is defined.    
- It is copied into **foreign keys** in other tables.
- It is often part of **indexes** used by the DBMS.
    
>Because of this, **shorter keys are preferred**. Smaller keys reduce:

- Disk storage
- Index size
- Memory usage during queries
    
***Example:***
Relation:  
Student(ID, Email, PassportNumber)

***Candidate keys:***
- ID (integer, 4 bytes)
- Email (string, maybe 30 bytes)
    
>Choosing **ID** as the primary key is better because it uses **less storage**, especially when repeated as foreign keys in many tables.

---

Key functionality within the database system

>This refers to how well the key works operationally in the database. A good primary key should be:

- Stable  
	The value should **not change over time**. If the key changes, every referencing foreign key must also change.

- Simple  
	Prefer **single attributes** instead of composite keys when possible. Simpler keys make joins and queries easier.

- System controlled  
	Keys generated by the system (for example auto increment IDs) are often better than real world attributes.

Meaningless  
	A key ideally should **not encode business meaning**. If business rules change, the key might need to change.

Example comparison:

Bad primary key:  
```
Email  
```

Reason: users may change their email.

Better primary key:  
```
StudentID
```

(generated integer)

---

Restrictions:
- It can NEVER be NULL.
- It can never be repeated, it is unique for each row.
- It must be "static", meaning it cannot change over the course of time.

### Composite primary keys:

>A composite key is a primary key made from two or more columns that together uniquely identify each record in a table. Individually, these columns may not be unique, but their combined values ensure uniqueness. Composite primary keys may be confused with candidate keys ([[Claves candidatas vs. claves primarias compuestas]]).
